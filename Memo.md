# 기록용 메모

## latin -> euc 계열
EUC-KR은 한글 1자를 2바이트로 표현하지만 latin-1은 모든 문자를 1바이트로 표현함(한글 지원 X)

따라서 latin-1에선 euc-kr 문자열의 한글 1자를 라틴어계열 2문자로 인식하여 읽어낸다.


CP949는 EUC-KR에서 지원하지 않는 한글 문자열을 추가한 확장형으로, latin-1과는 위와 동일한 문제를 가진다.


### 1. latin-1 -> euc-kr
 1. 오류 텍스트를 latin-1 바이트로 변환
 2. 변환된 바이트를 euc-kr 형식으로 다시 디코딩

### 2. latin-1 -> CP949(0xAD 치환)
 1. 오류 텍스트를 latin-1 바이트로 변환
 2. 변환된 바이트에서 0x5F 바이트('_')를 찾아서 0xAD 바이트로 치환
 3. 수정된 바이트를 CP949 형식으로 다시 디코딩
 
    #### 단점 
    모든 오류 바이트를 0xAD로 치환하는 방식이 모든 문자열에서 유효한지 확인되지 않음. 

    오류 바이트가 아닌, 원본 데이터에 있는 언더바가 소실됨 (추후 수정 예정)


### 3. latin-1 -> CP949(치환 없음, 원본 텍스트에 언더바가 있는 경우)
 1. 오류 텍스트를 latin-1 바이트로 변환
 2. 변환된 바이트를 CP949 형식으로 다시 디코딩

단점 : 망가진 바이트까지 그대로 리턴하기에 문자열 일부 깨짐

## UTF-8 -> CP949 계열
오류 문자가 '�' 또는 '_'로 치환되어어있음. (운영체제 차이?)

UTF-8에선 한글 1자를 표시하기 위해 3바이트를 사용하지만 CP949에선 1자당 2바이트를 사용함.

따라서 UTF-8 바이트를 CP949 형식으로 읽어버리면 2바이트씩 나눠서 읽기에 마지막 1바이트는 뒤쪽 문자 바이트로 인식되어진다.

그중 CP949 인코딩 방식에 사용되지 않는 조합이 생성될 경우 존재하지 않은 문자로 인식되기에 '�' 또는 '_'로 덮어씌워진다. (원본 바이트 소실)

latin-1 <-> euc-kr(cp949)은 1자마다 1바이트씩 읽어내어 오류를 판별하기에 오류 문자가 적은 반면, CP949<->UTF-8은 한 글자를 이루는 2개의 바이트 모두 기준에 만족하지 않으면 2바이트 전부 오류 문자로 바꿔버리기에 소실되는 바이트가 많다.
